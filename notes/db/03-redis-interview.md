## Redis 的应用场景、优缺点

### 常见场景：
- **缓存**：减轻数据库压力，提高读性能
- **实时统计**：计数、UV、签到、活跃用户统计
- **消息协作**：消息队列、延迟处理、发布订阅
- **分布式系统共享状态**：分布式锁、限流、会话共享

**电商秒杀**

- 用 `String / Hash` 预存库存
- 用 `DECR` 原子扣减库存
- 用 Redis 做限流，防止瞬时流量冲垮服务
- 用 `Set` 记录秒杀成功用户，防止重复请求

**社交媒体**

- 用 `List` 存 Feed 流、时间线、消息列表
- 用 `Set` 存点赞、关注、共同好友
- 用缓存加速热点内容访问

### 优点：

- **性能高**：基于内存，读写速度快
- **命令原子性强**：单线程执行命令，单条命令天然原子
- **数据结构丰富**：String、Hash、List、Set、ZSet、Bitmap、HyperLogLog 等
- **支持高可用和扩展**：可通过 Sentinel 和 Cluster 实现高可用与扩容

### 缺点：

- **数据可靠性不如关系型数据库**
    - RDB 可能丢失最后一次快照之后的数据
    - AOF 文件更大，恢复更慢
    - AOF 每秒刷盘模式下，极端情况下仍可能丢失约 1 秒数据
- **大 Key / 慢命令会阻塞**
    - 单线程模型下，长耗时命令会影响其他请求
- **缓存问题需要额外治理**
    - 缓存穿透
    - 缓存击穿
    - 缓存雪崩
    - 缓存一致性问题

## 常见缓存问题有哪些

### 1）缓存穿透
大量查询不存在数据库的key请求，请求每次都会打到数据库。

**产生原因：**

- 大量恶意请求不存在的key

**解决方案：**

- 参数校验，
    - 简单有效，但对合法不存在的参数无能为力
- 缓存空对象
    - 从数据库中查询不存在的key加入到缓存，value设置为nil，以及短ttl
    - 内存占用大，真实数据短时间无法被查询
- 布隆过滤器
    - 将所有查询条件预先加入到布隆过滤器
    - 过滤器中不存在，直接返回空结果，不走缓存和数据库
- 实时监控与黑名单
    - 网关层直接拦截
- 接口限流降级
    - 网关层或服务层用令牌桶、漏桶算法

---

### 2）缓存击穿
某个存在的热点 Key 过期失效并被移除时，大量并发请求这个数据，打到数据库，是雪崩的特例。

**场景：**

10点整，抢coding plan优惠券，旧的优惠券缓存刚过期，新的还没生成。

**解决方案：**

- 互斥锁
    - 第一个发现缓存失效时，不直接查数据库
    - 先尝试获取分布式锁，只有获得锁的请求线程才有资格查询数据库以及重建缓存
    - 其他未抢到锁的线程，自旋或者直接返回默认值
- 热点数据永不过期
    - 用value逻辑过期时间替代ttl，
    - 查询时判断当前时间是否早于value的逻辑时间
    - 如果是，直接返回数据
    - 如果不是，尝试获取分布式锁，只有获得锁的请求线程才有资格查询数据库以及重建缓存，其他未抢到锁的线程，直接返回旧值
- 提前续期热点数据
    - 设短ttl(5min)，后台异步刷新
    - 启动定时任务，key还剩1min过期时，主动查询数据库刷新缓存，重置过期时间
    - 适用于能预测热点key的场景，可以采用redission库，内置看门狗机制
    - 监控热点命令redis hotkeys
    - 引入本地缓存caffine

---

### 3）缓存雪崩
大量 Key 在同一时间过期，导致请求集中打到数据库。

**产生原因：**

- 大量key设置同一ttl，比如凌晨批量刷新
- Redis服务宕机
- 热点数据集中失效

**解决方案：**

- 过期时间加随机值
- 热点数据用逻辑过期时间替代ttl，查询时先返回旧数据，用一个线程异步更新缓存
- 数据预热与更新策略
    - 定时任务，缓存快失效时重新加载
    - 不依赖过期，数据变更时采用双写
    - 缓存降级，返回兜底数据
- 服务熔断限流，队列缓冲请求
- 高可用架构
    - 主从复制+哨兵
    - Redis Cluster，自动故障转移
    - 数据库也采用读写分离，请求分发到多个读库
- 监控告警
    - 缓存命中率低于80%告警
    - Redis CPU/内存使用率
    - 数据库连接数/QPS
- 快速响应机制
    - 降级开关
    - 一键切换本地缓存
    - 动态ttl配置中心

**完整流程：**

- 监控发现缓存命中率骤降，触发告警
- 自动启用本地缓存
- 限流组件限制数据库请求
- 服务降级返回默认数据
- 异步线程重建缓存
- 逐步恢复关闭限流

**场景选择：**

- 电商大促：数据预热+永不过期+限流
- 日常业务：随机过期+多级缓存
- 数据频繁更新：主动更新+版本控制
- 突发流量：熔断降级+队列缓冲

### 4）缓存一致性
- **产生原因**
    - 操作时序问题：数据库与缓存操作非原子性
    - 并发冲突：多线程操作同一数据
    - 网络延迟失败：一个成功，一个失败
    - 缓存与数据库结构差异：数据转换过程不一致
- **常见方案**
    - 旁路缓存（Cache Aside）弱一致性
        - 读操作：先从缓存读，若没有从数据库读，写入缓存
        - 写操作：查询数据库值，先更新数据库，再删除缓存
        - 存在不一致
            1. 删除缓存之前的时间段会有读请求读到旧缓存，类似可重复读
            2. 更新数据库完成之前的时间段内读请求没命中缓存从数据库读到了旧值；在整个操作之后回填了缓存。直到下一次更新数据库才有可能覆盖旧值或者缓存设置了ttl才能失效。
            3. 先删除缓存，再更新数据库也会有问题。删除缓存后更新数据库之前，读请求还是能回填旧数据。
    - 直写模式（Write Through）强一致性
        - 写操作：查询数据库值，同步改缓存和数据库
        - 由缓存组件实现，应用层只负责操作缓存，感知不到数据库
    - 异步回写（Write Behind）弱一致性
        - 写操作：查询数据库值，先改缓存异步改数据库
        - 数据库事务可能会回滚，不一致持续到下一次更改；内存、消息队列、Redolog之前系统崩溃导致数据未落库；并发冲突，后写覆盖先写同一个数据
- **高级方案**
    - 延迟双删解决Cache Aside的不一致问题
        - 写操作：先第一次删除缓存，更新数据库，延迟删除，异步延迟消息队列/线程池重试（这里是更新数据库时间内别的读请求再次读到旧数据并回填的缓存）
        - 缺点：
            - 这个异步时间如何精准控制（主从同步+业务读+安全缓冲）
            - 多个写操作并发问题
            - 第二次删除缓存时，Redis闪断，删除失败。
            - 数据库主从架构，读请求的从库还没同步仍然回填旧值
    - 基于版本号/时间戳的乐观锁
        - 读操作：
            - 读缓存，命中返回；未命中，查询数据库，执行Lua脚本尝试回填业务缓存
            - 先获取版本号缓存，若版本号不存在，直接写入
            - 若存在但数据库版本号落后缓存，说明读操作查询数据库之后有写操作更新了版本号缓存，静默不回填业务缓存
            - 若存在且版本一样，回填业务缓存
            - 无论缓存是否成功回填，返回数据库查询得到的数据
        - 写操作：
            - 先查询数据库得到数据和版本号，计算得到新数据，版本号递增
            - 更新数据库时比对版本/时间戳，若冲突抛异常。
            - 若成功则更新版本号缓存，删除业务缓存
        - 解决Cache Aside读请求回填旧数据的问题，因为读请求回填时会检查缓存版本号；同时写请求乐观锁保证了数据库的并发写安全，缓存只更新版本号。
        - 解决了延迟双删中写操作并发问题，只要写操作成功，版本号必然递增；任何小于版本号的读都不会更新缓存，杜绝了缓存层面的脏数据。
        - 主从架构数据库问题仍然无法解决，高并发场景，数据库写压力剧增
        - 数据库新增额外版本号字段
    - Binlog订阅方案-CDC(Change Data Capture)
        - 读操作先从缓存读，若没有从数据库读，设置一个ttl并写入缓存。
        - 写操作只更新数据库
        - CDC中间件（Canal）实时拉取解析变更前后数据，结构化为json发送到MQ
        - 缓存消费者读取MQ消息，根据消息中的key执行缓存删除
        - 优点：彻底解耦。Mysql Binlog顺序在并发时按照事务先后顺序，但提交顺序并不知道。Canal通过伪装为从库，按照主库binlog顺序提交给消息队列，消费者拿到后与Redis比对版本号/时间戳删除缓存，规避并发脏数据
        - 缺点：网络抖动，重复binlog被消费，删除天然幂等没事，但若引入乐观锁实现毫秒级最终一致性，需要更新版本号缓存，也是幂等也没问题。有个多线程消费者陷阱，必须保证同一key的消息在同一个partition保证顺序消费。否则后面的更新版本号就出现错乱，Lua脚本加入一个防倒退版本号判断更新兜底。
    - 分布式锁保证强一致
        - 写操作：先获取Redis分布式锁，再更新数据库，删除缓存，释放锁。
        - 读操作：查缓存，查数据库，回填缓存。
        - 整个操作为原子操作，读操作不会读缓存脏数据。
        - 仍然解决不了主从数据库架构问题
    
    **场景选择**
    
    强一致性：分布式锁+同步双写-金融交易、库存扣减
    
    最终一致性：Binlog (CDC)+MQ ****-商品信息、用户资料
    
    弱一致性：异步更新+过期时间-社交动态、推荐内容

## 热key 和大key问题发现与解决

### 热key问题

- 请求倾斜、单节点负载高、集群负载不均衡、影响其他业务访问

### 大key问题

- 内存分布不均、操作阻塞线程、网络拥堵、数据迁移失败

### 监控手段

- 内置命令扫描：redis-cli —bigkeys/hotkeys、redis-memory-for-key <key>、slowlog get <num>
- 代理层监控：埋点统计
- 网络流量抓包分析
- RDB控制台

### 热key解决方法

- 热key拆分子key，分片
- 本地缓存短ttl避免一致性问题
- 集群分片，对热key+随机后缀
- 为热key节点增加只读副本（从节点），客户端轮询访问不同副本
- 启用CDN缓存，智能DNS路由边缘节点，比如淘宝商品图片，抖音热门视频

### 大key解决方法

- 数据拆分
    - field拆分
    - hash分片
- 数据结构优化
    - 数据压缩
    - hyperloglog替代大set
    - bitmap替代boolean列表
- 异步删除过期key

## Redis 为什么设计为单线程模型

Redis采用单线程处理命令+多线程网络I/O+后台线程混合模型

### 避免并发复杂性

- 无锁设计，天然避免并发竞争
- 原子性保证，命令都是原子执行
- 可预测行，没有线程切换开销

### 性能优势

- 内存操作，纳秒级操作
- CPU缓存友好，局部性原理

### I/O多路复用

- 非阻塞I/O，epoll/kqueue机制，数万并发连接
- 事件驱动，Reactor模型，主线程只负责就绪事件

## Bitmap 如何实现布隆过滤器

### 实现原理

布隆过滤器底层可以用 Bitmap 实现。

**步骤如下：**

**1）初始化**

创建一个足够大的 Bitmap，初始值全为 0。

**2）添加元素**

对元素做多次哈希，得到多个位置，把这些位置都置为 1。

**3）查询元素**

对元素做相同次数的哈希，检查对应位置：

- **只要有一个位置是 0**：元素一定不存在
- **如果全是 1**：元素可能存在
- **为什么不会漏判**：因为相同元素hash后的位置都是相同的，那个位置一定得是1
- **为什么会误判**：因为别的元素可能和这个不存在的元素映射的位置重合

### 特点

- **优点**：空间效率高，查询快
- **缺点**：有误判率，且通常不支持删除

### 应用场景

- **防止缓存穿透**：先判断 ID 是否可能存在，不存在则直接返回
- **推荐系统去重**：过滤用户已经看过的内容
- **爬虫 URL 去重**：避免重复抓取
- **防止重复注册**：快速判断用户名/手机号是否可能已存在

### Redis 怎么支持布隆过滤器？

Redis 原生没有标准 Bloom 类型，但可以：

- 用 **Bitmap 自己实现**
- 使用 **RedisBloom 插件**

## HyperLogLog 原理

### HyperLogLog 是做什么的？

HyperLogLog 用来做**海量数据的去重计数**，也就是基数统计。

比如统计一天有多少独立访客（UV）。

### 核心思想是什么？

它不是直接保存每个元素，而是利用**概率统计**来估算基数。

步骤可以这样理解：

1. 把元素哈希成一个足够均匀的二进制数
2. 观察这个二进制数里**前导连续 0 的个数**
3. 前导 0 越多，说明这个样本越“稀有”
4. 样本越稀有，通常说明总体样本量越大

### 为什么前导 0 个数可以估算基数？

对于随机二进制数：

- 前导 1 个 0 的概率是 `1/2`
- 前导 2 个 0 的概率是 `1/4`
- 前导 k 个 0 的概率大约是 `1/2^k`

所以如果观测到了较大的前导 0 值，就说明样本规模可能已经比较大。

因此可以近似认为：

`N ≈ 2^k`

### Redis 中是怎么做的？

Redis 会把数据分到 **16384 个桶** 中：

- 哈希值的前 14 位决定桶号
- 后面的位用于统计前导 0 个数
- 每个桶记录自己见过的最大前导 0 值
- 最后对所有桶做调和平均，估算整体基数

### HyperLogLog 的优缺点

**优点：**

- 占用内存固定
- 特别适合海量去重计数

**缺点：**

- 有误差，不适合精确计数
- 不保存原始元素，因此**无法取回具体成员**
- 也不适合做精确删除

## Redis 分布式锁 / 自旋锁实现

### Redis 分布式锁怎么实现？

最常见方式：

```
加锁：SET lock_key unique_value NX PX 30000
解锁：先校验 unique_value，再删除 key
```

### 为什么不能只用 SETNX？

因为只 `SETNX` 还不够：

- 没有过期时间，可能造成死锁
- 解锁时如果不校验 owner，可能误删别人的锁

所以更推荐：

```
SET key value NX EX/PX ttl
```

### 解锁为什么要校验 value？

因为锁可能超时失效，随后被别的线程拿到。

这时旧线程再执行 `DEL`，就可能把别人的锁删掉。

所以必须保证：

- 只有加锁的线程
- 才能释放自己的锁

通常通过 Lua 脚本保证“比较 value + 删除”原子执行。

### 什么是锁续期？

业务执行时间可能超过锁的过期时间。

为了避免锁提前失效，需要在后台定时续期。

常见实现：

- WatchDog 机制
- 定时刷新过期时间

### 什么是自旋锁？

自旋锁指线程在获取锁失败后，不立即挂起，而是循环尝试获取锁。

**优点：**

- 避免线程频繁切换带来的开销

**缺点：**

- 如果锁长期拿不到，会浪费 CPU

### 自旋锁底层依赖什么？

底层通常依赖 **CAS（Compare And Swap）**：

- 比较当前值是否等于预期值
- 如果相等，则原子更新成新值
- 如果不相等，则操作失败，继续重试

CAS 是硬件级原子操作，常用于实现轻量级锁、自旋锁。

### `SETNX` 和 `SET NX` 有什么区别？

- `SETNX`：独立命令
- `SET NX`：`SET` 命令的一个可选参数

一般更推荐 `SET NX EX/PX`，因为可以在一条命令里同时完成：

- 不存在才设置
- 设置过期时间

## Redis 持久化

### RDB(Redis Database)

- **工作原理：定时生成内存数据的二进制快照**
- **触发方式：**
    - 手动SAVE（阻塞）与后台BGSAVE（异步）
    - 配置自动触发save 900 1，900s内至少一个key被修改
    - 执行SHUTDOWN / FLUSHALL命令
- **核心流程：**
    - fork()创建子进程，copy on write
    - 子进程将内存数据写入RDB文件
    - 替换旧RDB文件

### AOF(Append Only File)

- **工作原理：记录所有写操作命令，追加到文件缓冲区**
- **写入策略：**
    - always：每个写命令同步刷盘，大key write+fsync系统调用会阻塞
    - everysec：每秒同步一次，大key write复制到内核较慢
    - no：操作系统决定何时fsync
- **重写机制：**创建新AOF文件，移除冗余命令，解决AOF文件膨胀，大key会让AOF很快出现重写，fork时页表的复制开销大。

### 混合持久化（**MP-AOF**）

- 追求高性能，仅使用RDB
- 追求数据安全，仅使用AOF
- 平衡性选择混合策略，文件包含RDB格式数据+AOF命令
- 重写开始时，主进程打开incr AOF文件，新的写命令都写入这个文件
- 子进程负责RDB数据搬运，Redis原子更新manifest文件，指向新的base和incr

---

## Redis AOF重写过程

### 触发时机

- 自动触发，通过配置AOF最小检查文件体积和AOF文件增长率（对比上次文件体积）实现
- 手动触发，BG REWRITE AOF
- 启动时触发，若AOF文件损坏

### 执行过程

- fork创建子进程执行重写任务，避免阻塞主进程。这里为什么不是线程，因为fork进程的COW特性，天然内存快照；若用线程天然共享内存，会有已经写过的数据可能被修改。
- 扫描所有key-value，为每个key生成set命令，将所有命令写入temp AOF
- 重写期间，主进程新的写命令会同时写入AOF缓冲区（同步到旧AOF文件,防止丢失数据，但是默认配置是不同步）以及AOF重写缓冲区（已经重写到新AOF后又修改的情况）
- 子进程发起完成信号，父进程将AOF重写缓冲区命令追加到新AOF文件
- 原子替换AOF文件（rename系统调用，inode链接）

## Redis 内存淘汰策略

### NoEviction

- 默认策略，新写入数据直接报错OOM
- 适合数据一致性场景，不容许数据丢失

### allkeys-lru

- 从所有key中淘汰最近最少使用（Redis采用近似随机采样LRU）
- 适合热点数据分布均匀场景，保留常用数据

### volatile-lru

- 从设置了ttl的key淘汰最近最少使用
- 持久化数据常驻内存，临时数据淘汰

### allkeys-random

- 从所有key中随机淘汰
- 适合所有key访问频率均匀，没有明显访问规律场景

### volatile-random

- 从设置了ttl的key中随机淘汰
- 持久化数据常驻内存，临时数据随机淘汰

### volatile-ttl

- 从设置了ttl的key中淘汰剩余存活时间最短的key
- 优先淘汰即将过期的key，适合短期缓存场景

### allkeys-lfu

- 从所有key中淘汰最少频次使用
- 适合基于访问频率的场景，淘汰冷门数据

### volatile-lfu

- 从设置了ttl的key中淘汰最少频次使用
- 适合基于访问频率的场景，淘汰临时冷门数据

---

## Redis 过期策略（协同工作，一同启用）

### key写入策略

- 当key设置了ttl，会将这个key和ttl写入过期字典hashtable

### 惰性删除

- 当设置了ttl的key被访问时，先检查是否过期
- CPU资源消耗少， 但会造成内存泄露

### 定期删除

- 周期性从设置了ttl的key中随机抽查并删除过期key
- 若本轮删除的key比例大于25%，再来一轮
- 作为后台线程也要被CPU调度，内存数据会有残留，无法实时删除

## Redis 分布式锁原理

### 为什么需要分布式锁？

对于同一个业务的多次请求，可能被负载均衡打到其他节点。使得别的实例同时处理导致错误。

我们通过对业务id哈希固定路由到一个实例上，分布式问题就成了单机问题，而对实例内部启用本地锁或者singleflight模式。

singleflight：**在高并发场景下，将多个相同的请求合并成一个，只让其中一个请求真正去执行（比如查数据库、调API），其他请求等待并共享这个结果**

### SET Lock_key unique_value NX EX 30

- Redis 给每个请求生成唯一UUID，只有锁不存在时，才能加锁。抢占和设过期时间一步原子操作，抢占重试前得先GET保证幂等。
- 若业务时间较长，需要watch dog机制续锁。
- 单节点锁缺点
    - 一旦mater宕机，锁信息还没同步给从节点 升为主节点，两个请求都加锁成功。
    - Redisson检查到连接异常，同步重试lock()，阻塞Redis后续所有操作
- Redlock方案
    - Redisson客户端将请求的加锁操作同时发给多个master节点，只有超过半数master节点都加锁成功；
    - 若失败，根据UUID判断，和释放锁，这两步放在Lua脚本里，否则判断完后锁过期又被抢占，释放的是别人的锁。
    - watchdog机制不是万能，一旦锁服务器宕机，watchdog线程也关闭，锁仍然会过期。
    - 业务逻辑必须短小精悍

## Redis 主从复制原理

### 什么是主从复制？

Redis 主从复制是指：

- **主节点负责写**
- **从节点负责读**
- 主节点把数据同步给从节点

主要作用：

- 读写分离
- 故障恢复
- 提升可用性

---

### 全量复制过程

**触发时机：**

- 从节点第一次连接主节点
- 断线太久，无法继续增量同步

**流程：**

1. 从节点向主节点发起同步请求
2. 主节点执行 `BGSAVE`，生成 RDB 快照
3. 主节点把 RDB 文件发送给从节点
4. 在发送期间的新写命令，暂存在主节点为每个从节点分配replication buffer中
5. 从节点加载 RDB 到内存
6. 主节点再把复制缓冲区中的增量命令发送给从节点
7. 主从建立TCP长连接

---

### 增量复制过程

全量复制完成后，主节点会把后续写命令持续同步给从节点。

**核心机制：**

- 主节点维护复制积压缓冲区（replication backlog buffer）
- 从节点记录复制偏移量 offset
- 如果网络短暂中断，从节点重连后会带着 offset 请求继续同步
- 主节点判断 offset 是否还在积压缓冲区内：
    - **在**：走增量复制，通过replication buffer发给从节点
    - **不在**：退化为全量复制

---

## 主从复制与 Redis Cluster 的区别

### 主从复制解决什么问题？

- 数据冗余
- 高可用
- 读写分离

但它**不能真正解决单机写入瓶颈**，因为写请求仍然集中在主节点。

### 哨兵机制

哨兵节点建立：订阅某个频道发现哨兵集群

主节点宕机，需要哨兵来通知客户端新主节点地址。

哨兵如何选举leader来负责后续的主从切换？

- quorum(一般同majority)和majority=(n+1)/2
- 哨兵判断主节点主观下线后，向其他节点询问，若超过quorum，则判断主节点客观下线，给自己投一票，但对于其他节点的询问先来先到。需要majority票数才能选举成功；若这一轮没成功，则各个哨兵节点随机指数退避发起投票。

### 主从切换流程

- 选出从节点（优先级>Offset>id），其他从节点复制id改成该从节点的
- 新主节点IP地址和信息，通过Pub/Sub模式通知客户端
- 旧主节点上线时，降级为从节点

### Redis Cluster 解决什么问题？

Redis Cluster 主要解决：

- 单机容量上限，即便哨兵主从，从节点只是副本受到服务器内存影响
- 单机写入性能瓶颈
- 水平扩展问题

它通过**数据分片**把 Key 分布到多个主节点上。

### 数据分片如何实现？

关键字只需计算一次，绑定hashslot，后续增加或减少分片，只需整体移动hashslot，无需对关键字再次hash。解耦了key与机器，那么客户端（go-redis）如何发现找到节点，启动时拉取hashslot和节点的映射表，后续请求时就可以计算得出。当slot移动，下次请求节点会返回Moved命令告诉客户端新的分片；若请求时正在移动，原节点回复客户端ask转向，客户端ask得到结果。

### 集群脑裂问题

从节点下线也是主节点判断客观下线。采用奇数个主节点，当一个主节点下线时，其余主节点对该主节点的从节点投票选举大于一半。

### 两者的核心区别

| 对比项 | 主从复制 | Redis Cluster |
| --- | --- | --- |
| 目标 | 高可用、读写分离 | 高可用、水平扩展 |
| 数据分布 | 所有节点保存同一份主数据副本 | 数据分片存储 |
| 写入能力 | 主要依赖单主节点 | 多主节点分摊写压力 |
| 适用场景 | 读多写少 | 数据量大、写压力大 |

### Cluster 和主从复制是什么关系？

不是互斥关系。Cluster中也有主从，主节点分摊读写，从节点可以是热备不负责读写

**Redis Cluster 中，每个分片通常仍然是一主多从结构。**

也就是说：

- Cluster 负责**分片，每个cluster的主节点都会有个复制id**
- 主从复制负责**副本和高可用**

## Redis 脑裂问题

### 产生原因：

- 网络分区导致主节点与从节点、哨兵/集群节点网络断开
- 哨兵或集群认为主节点失效，选举出新主节点
- 原主节点在网络隔离后仍接受客户端写入

### 导致后果：

- 数据不一致：两个主节点同时写入数据，无法自动合并
- 数据丢失：恢复后，其中一个主节点数据会被覆盖
- 服务混乱：客户端连接到不同主节点，业务逻辑出错

### 解决方法：

- Redis Sentinel 模式
    - 合理配置
        - min-slaves-to-write：主节点必须有N个从节点连接才允许写入
        - min-slaves-max-lag：从节点复制延迟不超过该值被认为有效节点
        - quorum：判定主节点客观下线的共识哨兵节点数为哨兵数一半以上
- Redis Cluster 模式
    - 通信机制
        - 节点通过Gossip协议定期通信
        - 每个节点维护集群拓扑视图
    - 关键配置
        - cluster-node-timeout：节点超时时间
        - cluster-migration-barrier：故障转移平衡主节点的从节点数的机制
        - cluster-require-full-coverage：故障转移需要的从节点票数
- 通用解决方法
    - 合理超时配置：down-after-milliseconds>2*RTT，避免网络抖动误判
    - 部署架构优化
        - 哨兵/集群节点分散在不同物理机（服务器）、机架（机房）、可用区（逻辑分区如华东地区）
        - 至少3个哨兵节点
        - 主节点跨机架/可用区部署
    - 监控告警
        - 主从连接数
        - 网络分区情况
        - 复制延迟告警
- 脑裂后的解决方法
    - 暂定服务
    - 选择主节点
    - 重置从节点
    - 数据修复
    - 恢复服务
