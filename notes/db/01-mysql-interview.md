## Mysql查询语句执行流程

- Server层
    - 连接器、查询缓存、解析器、预处理器、优化器、执行器
    - 内置函数、存储过程、触发器、视图
- 存储引擎层
    - innodb，索引
- 执行流程
    - TCP三次握手→校验用户名密码→读取权限
    - key为查询语句，value为查询结果(8.0之后就没有这一层了)
    - 词法分析、语法分析、构建语法树AST
    - 检查表、字段是否存在；将*改为所有字段
    - 根据索引的查询成本估算，若explain里的执行计划没有key，执行全表扫描
    - 执行
        - 主键索引查询：存储引擎通过主键索引B+树定位第一条记录，判断是否符合查询条件
        - 全表扫描：存储引擎从表的第一条记录开始，若每符合一条记录就直接发给server层
        - 索引下推：
            - 无下推：先查询满足第一个查询条件的记录，获取主键值，然后回表查询主键索引B+树，最后交给server层判断是否满足其他查询条件
            - 有下推：先查询满足第一个查询条件的记录，判断联合索引中后续字段是否满足查询条件，满足才交给server层
        

## Mysql更新语句执行流程

- 执行器调用存储引擎接口，通过索引树找到记录，若该行记录在Buffer pool中直接交给执行器，若不在从磁盘读到buffer pool，返回记录
- 执行器判断更新前后有无变化，若无直接返回；若有则调用存储引擎接口，参数为新旧记录
- 开启事务，生产UndoLog，写入buffer pool的Undo页面，将修改记录写入RedoLog
- 更新内存，标记脏页，将记录写入RedoLog，记录Binlog保存到binlog cache
- 事务提交两阶段（内部XA事务）
    - binlog作为协调者，存储引擎作为参与者
    - prepare阶段
        - XID写入RedoLog，事务状态设置为prepare，落盘
    - commit阶段
        - XID写入BinLog，落盘，调用存储引擎接口提交事务，把RedoLog状态改为Commit（在page cache里改，不急于直接落盘）
    - 问题
        - 性能开销大，每个事务提交两次fsync
        - 多事务下，需要加锁控制事务依次提交，保证两个日志提交顺序一致。
    - 解决方法
        - Binlog引入组提交
            - 将多个事务的binlog刷盘操作合并为一次，prepare阶段不变
            - commit阶段3个队列，每个队列引入锁控制并发
                - flush：将binlog按进入顺序写入page cache
                - sync：fsync落盘
                - commit：各个事务按顺序调用存储引擎commit

## Mysql异常重启流程

- Redo Log
    - 扫描最后一个Checkpoint之后的Redo Log
    - 将所有修改应用到Buffer Pool（磁盘上可能有未提交的数据，因为Mysql允许未提交数据刷盘，提升性能，Buffer Pool容量有限）
- Bin Log
    - 对于每个提交状态为prepare的RedoLog，用XID查找Binlog
    - 若无，说明Binlog未刷盘，回滚事务
    - 若有，则提交事务
- Undo Log
    - 上述需要回滚的事务读取UndoLog记录
    - 将未提交事务的修改回滚到修改前状态

## Mysql数据存储

- 文件结构
    - db.opt 默认字符集和配置
    - frm表结构+ibd表数据（每个表一份）
    - ibd结构
        - 记录按行存储，多行组成一页，数据库以页为单位读写
        - 每页大小16KB，包括数据、undo日志、溢出页等等
        - 当表中数据量大时，按区分配，使得B+树每一层双向链表的链表相邻的页物理位置连续。
        - 段分为数据段（叶节点）、索引段（非叶节点）、回滚段
    - 行结构
        - 变长字段长度列表、NULL值列表、记录头信息（delete mark、next record ptr、record type：0-普通；1-非叶；2-最小；3-最大）、row ID（没有主键时启用）、trx ID（事务id）、roll Ptr（上版本指针）、字段值
    - 页结构
        - 文件头、页头、最大最小记录、用户记录、空闲空间、页目录、文件尾
        - 记录按照主键顺序组成单向链表，页目录存储每组最后一条记录的地址偏移量（槽slot），通过二分法快速定位槽，再遍历该槽的记录组就可找到记录。

## Mysql索引

- 数据结构分类：B+、Hash（innodb无这个，做不了范围查询）、Full Text
- 物理存储：聚簇（主键）、二级（辅助）
- 字段特性：主键、唯一、普通、前缀
- 字段个数：单列、联合
- B+树
    - 主键索引树，非叶存索引，叶子存数据
    - 二级索引树，非叶存二级索引，叶子存主键
    - 若查询的字段在二级索引之中（包括主键），就算覆盖索引，即不用回表（二次查主键索引树）
- 聚簇索引默认主键，若没有主键，采用没有NULL值的列；若还是没有，则隐式生成
- 前缀索引是对字段的前几个字符做索引，减少索引占用空间
- 联合索引遵循最左匹配原则
    - 因为B+树节点内的排序是按照最左排序的，联合索引的后续关键词是局部顺序，不是全局顺序，所以范围查询后的等值查询就无法利用索引定位了
    - 索引区分度越大的字段应该排在前面，利用联合索引排序字段可以避免ORDER BY提高查询效率
- 范围查询可以利用联合索引，但是范围查询字段的后面字段就无法利用
- 索引下推可以在联合索引中包含的字段先判断，过滤掉不符合条件的记录，减少回表
- 优化方法
    - 前缀索引优化
        - 减少索引字段大小，增加索引页的索引数量，提高查询效率
        - order by无法使用；无法用作覆盖索引，导致回表
    - 覆盖索引优化
    - 主键自增
        - 若新插入随机索引，插入页中，导致页分裂，内存碎片。
- 索引失效
    - 在索引字段上做了计算，索引存储原始值，而非计算后的值
    - 左模糊，like %xxx，类似联合索引
    - or左边是索引，右边不是，where xx or yy，全表扫描没有索引的条件
    - 联合索引不满足最左匹配
    - 使用≠，不是等值查询全表扫描
    - 使用is not null
    - 隐式类型转换 where phone = num，但phone原本是varchar相当于做了函数计算
- 执行计划
    - key实际索引、rows扫描行数、file sort排序、temporary临时表聚合

## Mysql深度分页问题

- 问题1:由于mysql数据的crud后会导致查询10000条数据之后的10条起始id并不知道是多少，所以查询优化器要么选择使用B+树的索引双向链表顺序扫描，要么选择全表扫描
- 问题2:顺序扫描也无法计数到起始id才开始取出数据，因为通常这种分页查询都伴随着查询条件和排序，必须把前面的所有数据都取出，处理排序，处理NULL值等查询条件才能返回符合要求的10条数据
- 但无论怎么CRUD，由于主键自增，新的数据只会在上次查询ID之后。
- 解决办法
    - 记录上一页最后一条的索引id，因为链表或者全表的这次扫描的id一定大于上次最后查到的id。
    - 覆盖索引子查询+外查询通过主键获取整行数据
    - 限制用户翻页深度
    - 采用ElasticSearch
    - 具体流程
        - 后端在查询第一页数据时，将最后一条的cursor返回给前端
        - 前端下次查询时，附带cursor告知后端
        - 采用复合cursor{ created_at , id }
            - created_at可能重复
            - 查询条件和排序方向一致
        - 后端sql语句带上cursor
        
        ```jsx
        SELECT id, title, created_at
        FROM posts
        WHERE user_id = ?
          AND (
            created_at = ? AND id < ?
          )
        ORDER BY created_at DESC, id DESC
        LIMIT ?
        ```
        

## Mysql事务

- ACID
    - Atomicity：事务若失败，那么回滚后就像没发生
    - Consistency：无论怎么变化，数据库完整性约束得保证
    - Isolation：多个并发事务执行时有独立的完整数据空间，也不影响其他数据
    - Durability：事务的成功对数据的修改是持久性的
- InnoDB如何保证
    - 原子性通过UndoLog
    - 持久性通过RedoLog
    - 隔离性通过MVCC
    - 一致性通过上三者
- 并发事务问题
    - 脏读
        - 现象：事务读到了另一个未提交事务修改的数据
        - 问题：若事务回滚，原事务基于的数据操作是错误事实
    - 不可重复读
        - 现象：同一事务内，多次读同一数据的结果不同（后续读到了另一个事务提交的数据）
        - 问题：那事务基于该数据的操作逻辑混乱；但这个问题基于场景来看，若是追求实时性，是允许的
    - 幻读
        - 现象：事务内查询记录数量，前后两次不一样（当中有事务提交了插入或删除数据的事务）
        - 问题：比如该事务是插入不允许重复的数据结果被删除了；基于一个时间点的统计数据失效；查询数据不存在，但查询条件有数量限制，并发导致超卖
- 事务隔离级别
    - 读未提交
        - 允许事务数据变更未提交时就能被其他事务看到
        - 问题：脏读、不可重复读、幻读
        - 读操作 不加锁，写操作加行写锁
    - 读提交
        - 允许事务数据变更提交后能被其他事务看到
        - 问题：不可重复读、幻读
        - 通过每个语句执行前Read View
        - 读操作 MVCC+写操作加行写锁
    - 可重复读
        - 事务启动到结束看到的数据是同一份快照
        - 问题：幻读
        - 通过事务开始前Read View
        - 读操作 MVCC + 写操作 间隙锁 (Gap Lock）
    - 串行化
        - 对记录上读写锁，事务必须串行执行
        - 读操作 行读锁 + Next-Key Lock + 表级意向锁
- Read View
    - 关键字段
        - creator_trx_id ：创建该Read View事务的id
        - m_ids：创建该Read View事务时，活跃但未提交的事务id
        - min_trx_id：m_id最小值
        - max_trx_id：下一个要分配的事务id
        - 聚簇索引记录有两个隐藏字段：trx_id(事务改动生成),roll_ptr(旧版记录UndoLog)
        - created_time：用于UndoLog清理，通过比较undolog 的trx_id与系统中最早创建的活跃Read View的min_trx_id，即可清理。
        - inno DB维护
            - 活跃事务id，事务开始时加入，事务提交时移出
            - 分配新事务id时，先分配现在的max，然后max++
            - 全局Read View链
    - 访问流程
        - select前创建Read View，根据这个Read View，从roll ptr依次查询各个版本记录的trx_id。
        - 若creator_id==trx_id，自己记录可见
        - 若trx_id<min_trx_id，表示这个版本记录是由创建Read View之前就已经提交的事务生成的，代表该版本对当前事务可见
        - 若trx_id≥max_trx_id，表示这个版本记录是由创建Read View之后启动的事务生成的，代表该版本对当前事务不可见
        - 若min_trx_id≤trx_id<max_trx_id，表示事务记录在创建Read View之前就分配了。
            - 若trx_id在m_ids中，表示生成该版本记录的事务在创建Read View仍然活跃（未提交），该版本对当前事务不可见；
            - 若trx_id不在m_ids中，表示该事务id虽然已经分配，但在创建Read View时已提交（min只代表未提交的最小id，比这id大的事务已经提交了但也在范围内），该版本对当前事务可见。
- MVCC（乐观控制）
    - 虽然Mysql默认隔离级别上可重复读，也能很大程度避免幻读。
    - 快照读：select 通过MVCC解决幻读
    - 当前读：select update/insert/delete通过记录锁+间隙锁（范围内id无法插入记录）解决幻读
    - MVCC不能完全解决幻读，比如先快照读 插入记录 再当前读

## Mysql锁（悲观锁）

- 全局锁：mysqldump避免全局读锁备份，使用可重复读快照
- 表级锁
    - 表锁：本线程只能读/写该表的数据，不能读/写其他表。读锁其他线程可以读。写锁是独占，读写互斥，写写互斥。
    - 元数据锁(MDL):
        - 用于保护其他线程对该表结构做变更
        - CRUD时自动加MDL读锁；表结构改变时自动加MDL写锁
    - 意向锁
        - 作用：快速判断表内是否有行锁
        - 加读锁之前加意向读锁
        - 加写锁之前加意向写锁
        - 意向读写锁之间不冲突，但会和表锁冲突
        - 引入原因：**行锁和表锁**之间读写互斥、写写互斥。
        - 若无意向锁，加表写锁之前就要遍历所有行是否存在写锁。
        - 若有意向锁，其他线程直接查该表的意向写锁就可获知是否有行锁
    - Auto-Incr锁
        - 插入数据时，加上表级锁，插入语句执行完就立刻释放，无需等到事务提交。从而保证表插入数据是线性递增的。
        - 为提升性能，innodb提供了轻量锁，不用等插入语句执行完就可释放，但结合binlog的statement模式，会导致主从的插入顺序不一样，导致数据不一致，因此将binlog改为raw模式，使用主库的自增顺序就可以。
- 行锁
    - Record 锁
        - 只有读读不互斥
    - Gap 锁
        - 间隙锁之间不互斥，因为两个事务同时持有相同范围不影响自身的目的，只不过它们之间的行锁有冲突
        - 加锁时机：RR隔离级别，当前读除存在等值索引情况都要加比如范围查询，查空记录，非唯一索引
        - 目的：防止幻读，别的事务在记录间隙中插入新记录
    - Next - Key 锁
        - Record+Gap，范围内不能插入，指定行不能修改。next-key之前可能互斥，因为record写锁之间互斥
    - 插入意向锁
        - 插入前判断该位置是否被gap锁包含，若有则阻塞，阻塞期间生成插入意向锁
        - 实现多个事务之间有序等待队列，避免死锁
- Mysql加锁
    - Serializable隔离级别，自动加共享读锁，其余隔离级别不加读锁
    - 加锁对象是索引，基本单位是next-key lock，锁住等值部分，以及左/右扫的第一条记录，innoDB 的索引是有序的，范围查询需要稳定的扫描范围，防止索引页分裂或重排
    - 唯一索引等值查询更新，记录存在，退化为record锁，同时加上表级意向锁
    - 唯一索引等值查询更新，记录不存在，退化为gap锁，(上一条记录id，该记录id]
    - 唯一索引范围查询，等值查询部分，则用上述规则
    - 范围查询一定要有索引，否则全表扫描相当于把整个表锁住
- 避免死锁
    - 设置事务等待锁的超时时间
    - 开启主动死锁检测回滚事务

## Mysql日志

- UndoLog
    - 插入记录时，记录该记录的主键id
    - 删除记录时，完整记录，但是会先打上flag，purge线程完成真正的物理删除
    - 更新记录时，记录旧值，若该列不是主键，undo直接记录反向操作；若是主键列，先删除再插入整条记录
    - 清理由后台线程定期清理
- RedoLog(WAL技术)
    - 事务修改时，会写UndoLog，RedoLog中得记录UndoLog页面和数据脏页的修改，先将RedoLog写入RedoLog Buffer，事务提交时，将RedoLog落盘，RedoLog是追加循环写，顺序写比随机读写性能好，因此落盘性能比直接落盘数据页性能高
    - 刷盘时机
        - 事务提交时（可以修改参数）
            - fsync（默认，最安全）
            - 写到os page cache 由OS决定何时刷盘
            - 每秒后台线程fsync
        - Mysql正常关闭
        - Buffer空间被用了一半以上
    - RedoLog清理
        - 当事务的修改被正确落盘后，RedoLog就没用了。
        - RedoLog采用两个文件的页面组成环形队列循环写，head指向最旧未落库记录，tail指向最新未落库记录
- BinLog
    - 写过程：每个线程一个binlog cache，每一条更新sql操作就会写入cache
    - 事务提交前，将完整cache写入binlog文件（page cache），之后清空binlog cache
    - 刷盘策略
        - fsync（默认，最安全）
        - 写到os page cache 由OS决定何时刷盘
        - 每N次提交事务后fsync
    - binlog与redolog区别
        - binlog是server层日志，所有存储引擎都可以用
        - redolog是物理日志，记录表空间页面偏移量做了什么修改
        - binlog是append写，写满一个new一个；redolog是循环写
        - binlog用于备份和主从复制；redolog用于故障恢复
        - 删库就可以用binlog恢复
    - 格式
        - statement（逻辑日志）：记录每一条更新sql，但是主从执行uuid可能并不同
        - raw（默认）：解决动态函数问题，记录每一条记录变化，binlog文件会很大
        - mixed：结合二者优势，动态调整
- BufferPool
    - 从磁盘读取后修改先暂存在内存提高读写性能，缓存包括数据页、索引页、UndoLog页、插入缓存
    - 脏页都在Flush链表中，后台线程就可以遍历落盘
    - 缓存命中率通过改进LRU算法，将链表划分为两半区域，预读页会进入后半部分，页被访问并且old停留时间大于1s时进入前半部分，解决大批量数据进入buffer导致原本热数据失效挤入old
    - 淘汰页若是脏页会被落盘
- Double Write Buffer
    - 数据库页大小：16KB；操作系统页大小：4KB
    - 写一个数据库页需要写四次
    - 为了避免直接修改磁盘中的数据页过程中宕机，导致磁盘数据页失效
    - 设置内存中缓冲区，将数据脏页(不是redolog)写入 Double Write Buffer
    - fsync() 持久化到磁盘
    - 从 Double Write Buffer 写入实际位置
    - 若上一部崩溃，重启后用取出磁盘中的double write buffer还原
