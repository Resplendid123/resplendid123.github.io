> **考纲：客户/服务器模型、P2P模型、域名系统DNS（域名空间、域名服务器、域名解析过程）、文件传输协议FTP（工作原理、控制连接与数据连接）、电子邮件（组成结构、格式与MIME、SMTP与POP3）、万维网WWW（概念、组成结构、HTTP）**

## 客户/服务器模型：
{: #section-1 }

客户必须知道服务器的地址，而服务器启动后被动等待客户请求，无需知道客户的地址。

**应用：**

Web、FTP、远程登录、电子邮件等。

**特点：**

网络中各计算机地位不等，服务器可通过对用户权限控制达到管理客户机的目的。客户机之间不直接通信，可扩展性不佳，服务器能支持的客户机有限。

## P2P模型：
{: #section-2 }

服务器性能好坏决定了系统的性能，因此P2P模型的思想是网络中传输的内容不再保存在服务器上，每个结点都具有上传、下载功能，权利义务大体对等。任意一对计算机称为对等方peer，直接相互通信。本质上，每个结点既是客户也是服务器。

**优点：**

可将任务分配给多个结点，减少服务器计算压力，提高系统效率和资源利用率。

多个客户机之间共享文档，支持百万用户同时工作。

可扩展性好，网络健壮性强，单个结点失效不会影响其他结点。

**缺点：**

同时提供服务和获取服务，会影响整机速度；P2P程序使网络变得很拥塞。

## 域名系统DNS：
{: #section-3 }

域名系统用来将特定含义主机域名转换为IP地址，采用客户/服务器模型，运行在UDP-53端口，与IP地址不是一对一关系，可以一对多也可以多对一。

### 层次域名空间：

**域名：**

因特网采用层次树状结构命名方法，任何连接到因特网上的主机或路由器，都有一个唯一的层次结构名称，即域名。域是名字空间可以被管理的划分，有顶级域、二级域、三级域等。各级域名由其上一级域名管理机构管理。

每一级域名都由英文字母和数字组成，不超过63个字符，不区分大小写。最低级域名在左边，最高级在右边。完整域名不超过255个字符。

**顶级域名TLD分类：**

**国家顶级域名（nTLD）：**

如.cn中国.us美国.uk英国

**通用顶级域名（gTLD）：**

**如：com（公司企业）、net（网络服务机构）、org（非营利性组织）、int（国际组织）、edu（美国教育机构）、gov（美国政府部门）、mil（美国军事部门）。**

**基础结构域名（arpa）：**

用于反向域名解析，即IP地址反向解析为域名。

![]({{ '/assets/notes/cn/image096.png' | relative_url }})

### 域名服务器：

域名和IP地址的映射关系必须保存在域名服务器中，供所有其他应用查询。显然不能将所有信息都储存在一台域名服务器中。DNS使用分布在各地的域名服务器来实现域名到IP地址的转换。

**根域名服务器：**

根域名服务器是最高层次的域名服务器。每个根域名服务器都知道所有的顶级域名服务器的域名及其IP地址。世界上一共13个根域名服务器，由分布在各地的服务器集群组成。当本地域名服务器向根域名服务器发出查询请求时，路由器就把查询请求报文转发到离这个DNS客户最近的一个根域名服务器。

**顶级域名服务器：**

顶级域名服务器负责管理在该顶级域名服务器注册的所有二级域名。当收到DNS查询请求时就给出相应的回答，可能是最后的结果，也可能是下一级权限域名服务器的IP地址。

**权限域名服务器：**

权限域名服务器负责管理某个区的域名。每一个主机的域名都必须在某个权限域名服务器处注册登记。因此权限域名服务器知道其管辖的域名与IP地址的映射关系。另外，权限域名服务器还知道其下级域名服务器的地址。许多权限域名服务器同时充当本地域名服务器。

**本地域名服务器：**

本地域名服务器不属于上述的域名服务器的等级结构。当一个主机发出DNS请求报文时，这个报文就首先被送往该主机的本地域名服务器。本地域名服务器起着代理的作用，会将该报文转发到上述的域名服务器的等级结构中。它有时也称为默认域名服务器，本地域名服务器可能在同一个局域网中。本地域名服务器的IP地址需要直接配置在主机中的DNS地址。

### 域名解析过程：

客户端需要域名解析时，通过本机的DNS客户端构造一个DNS请求报文，以UDP数据报方式发给本地域名服务器。

1.主机发送ARP请求，询问DNS服务器的MAC地址（如果之前没有缓存）。

2.一旦获取到DNS服务器的MAC地址，主机就会构造一个目的MAC地址是DNS服务器的MAC帧。

3.主机将DNS请求发送到DNS服务器。

**递归查询：**

主机向本地域名服务器的查询都采用递归查询。本地域名服务器不知道被查询域名的IP地址，就递归替代主机查询。

![]({{ '/assets/notes/cn/image097.png' | relative_url }})

要注意域名中有多个权限域名服务器，需要多次请求，例如dns.abc.edf.com。

**迭代查询：**

本地域名服务器向其他域名服务器采用递归查询或迭代查询。本地域名服务器只需向根域名服务器查询一次，根域名服务器会告诉IP地址或者下一个顶级域名服务器的IP地址。

![]({{ '/assets/notes/cn/image098.png' | relative_url }})

**优化：**

为了提高DNS的查询效率，并减轻根域名服务器的负荷和减少因特网上的DNS查询报文数量，在域名服务器中广泛地使用了高速缓存。高速缓存用来存放最近查询过的域名以及从何处获得域名映射信息的记录。用户主机中也在启动时从本地域名服务器下载域名和IP地址的全部数据库，维护存放自己最近使用的域名的高速缓存，并且只在从缓存中找不到域名时才向域名服务器查询。

## 文件传输协议FTP：
{: #section-4 }

**特点：**

FTP提供交互式的访问，允许客户指明文件的类型与格式（如指明是否使用ASCII码），并允许文件具有存取权限（如访问文件的用户必须经过授权，并输入有效的口令），以匿名方式提供文件共享。修改文件必须先拷贝回主机修改后再上传，NFS可以解决这一问题。

屏蔽了各计算机系统的细节，适合在异构网络中任意计算机之间传送文件。

**工作方式：**

采用客户/服务器工作方式，使用TCP可靠传输服务。FTP服务器进程由两部分组成，一个负责接收新的请求，若干个负责处理单个请求。

FTP工作时使用两个并行TCP连接，一个是控制连接，端口号21；一个是数据连接，端口号20。

### 控制连接：

服务器监听21端口，等待客户连接，负责连接请求，传送请求等控制信息；控制连接在整个会话期间一直保持打开状态，是持久连接。

### 数据连接：

客户通过临时端口连接到服务器21端口，登录成功要传送数据时，就会在另一个端口创建数据传送进程。服务器每次在传送完毕后，就会释放连接，是非持久连接。

**数据传输方式：**

由于FTP采用分离的控制连接，也称FTP的控制信息是带外传送的。要修改服务器上的文件，需先将文件传送到本机，修改后再将副本上传。

**主动方式PORT（默认方式）：**

客户端随机开放另一个端口，通过控制连接发送PORT命令告知服务器，服务器收到PORT命令和端口号后，主动通过20端口和该开放端口建立数据连接。

**被动方式PASV：**

客户端随机开放另一个端口，通过控制连接发送PASV命令告知服务器，服务器在本地随机开放一个端口，并通过控制连接告知客户端端口号，客户端再主动连接到服务器开放端口进行数据传输。

## 电子邮件：
{: #section-5 }

### 过程：

1.是一种异步通信方式，发件人将邮件发送到自己使用的邮件服务器；

2.发件人的邮件服务器将收到的邮件按其目的地址转发到收件人邮件服务器中的收件人邮箱；

3.收件人在方便的时候访问收件人邮件服务器中自己的邮箱，获取收到的电子邮件。

### 用户代理

是用户与电子邮件系统的接口，又称为电子邮件客户端软件，如Outlook和Foxmail。

### 邮件服务器

同时充当客户和服务器，其功能是发送和接收邮件，同时还要负责维护用户的邮箱。

### 协议

邮件发送协议（例如SMTP）用于用户代理向邮件服务器发送邮件或者邮件服务器之间发送。
邮件读取协议（例如POP3，IMAP）用于用户代理从邮件服务器读取邮件。

![]({{ '/assets/notes/cn/image100.png' | relative_url }})

### 电子邮件格式：

分为信封和内容两大部分，邮件内容又分为首部和主体部分。用户写好首部后，邮件系统自动将所需信息提取写到信封上。

**首部：**

包括To、Subject、From、Date。To是必填关键字，后面填一个或多个收件人邮件地址。From也是必填关键字，通常由邮箱系统自动填写。

**邮箱名：**

必须是邮箱所在主机的域名里唯一的，即保证全网唯一，格式：用户名@邮箱所在主机域名。

### 多用途因特网邮件拓展（MIME）：

**SMTP缺点：**

SMTP协议只能传送7位ASCII码文本明文数据，不能传送可执行文件或其他的二进制对象。SMTP不能满足传送多媒体邮件（图片、音频或视频数据）的需要。并且非英语国家的文字也无法用SMTP传送。

**引入：**

为解决SMTP传送非ASCII码文本的问题，提出了多用途因特网邮件扩展 MIME。当邮件含有非ASCII码数据时，需要用MIME转换成ASCII码数据，再用SMTP传送，接收端需要用MIME逆转换。

**MIME特点：**

**1.增加了5个新的邮件首部字段，这些字段提供了有关邮件主体的信息。**

**2.定义了许多邮件内容的格式，对多媒体电子邮件的表示方法进行了标准化。**

**3.定义了传送编码，可对任何内容格式进行转换，而不会被邮件系统改变。**

### 简单邮件传送协议SMTP：

负责发送邮件的是SMTP客户，负责接收邮件的是SMTP服务器，采用TCP连接，端口号25。SMTP不使用中间邮件服务器，TCP连接都是发送方和接收方的邮件服务器之间直接建立，无论需要经过多少路由器。

**连接建立：**

发件人将邮件发到邮件服务器缓存中后，发送方SMTP客户周期性对邮件缓存扫描，若发现有邮件，就与接收方SMTP服务器建立TCP连接，服务器端口号25。连接建立后，SMTP服务器发出220 Service Ready服务已绪。SMTP客户向SMTP服务器发送HELO命令，附上发送方的主机名。

**邮件传送：**

从MAIL命令开始，附有发件人邮箱地址。若SMTP服务器准备好接收邮件，则回答250 OK，SMTP客户紧跟发送一个或多个RCPT命令，附有收件人邮箱地址。每发送一个命令，就有相应信息从SMTP服务器返回，如250 OK或550 No Such user here。获得OK后，客户端用DATA命令，表示要开始传送，服务器若回答354 Start Mail input，则SMTP客户就开始传送邮件。

**连接释放：**

客户发送QUIT命令，若服务器发送221 Closed信息，则表示同意释放连接。

![]({{ '/assets/notes/cn/image101.png' | relative_url }})

### POP3：

邮局协议POP是一个邮件读取协议，采用C/S模式，TCP连接，端口号110。接收方的用户代理必须运行POP客户程序，接收方的邮件服务器运行POP服务器进程。客户端提供用户名和口令才可读取邮件，可以选择下载并保留或删除（是否保存在服务器）。不允许用户在邮件服务器上管理邮件，密码是明文。协议基于ASCII码，非ASCII码需要MIME转换。

### IMAP：

因特网报文存取协议IMAP较之复杂，采用C/S模式，TCP连接，端口号143。为用户提供了创建文件夹，移动邮件，远程文件夹查询等联机命令，维持了会话用户的状态信息，密码是明文。它允许用户代理只获取报文的某些部分，用户可能并不想取回所有邮件，比如音频和视频等。客户端与邮箱更新同步。

### 基于WWW的电子邮件:

目前出现了很多基于万维网的电子邮件，如Hotmail、Gmail等，用户浏览器与邮件服务器之间的邮件接收和发送用HTTP，仅在邮件服务器之间用SMTP。用户无需安装用户代理程序，也无需将邮件下载到本地即可管理。

## 万维网WWW：
{: #section-6 }

万维网WWW并非某种特殊的计算机网络。它是一个大规模的、联机式的信息存储空间，是运行在因特网上的一个分布式应用。浏览器是主机上的万维网客户程序，而万维网文档所在的主机则允许万维网服务器程序。用户通过浏览器用URL与Web服务器建立连接，服务器把URL转换为文件路径，返回信息。

**组成：**

**统一资源定位符URL：**

是对因特网上可得到的资源的位置和访问方法的简洁表示，形式是\<协议\>://\<主机\>:\<端口\>/\<路径\>。协议如http、ftp；主机是域名或IP地址；端口和路径有时可以省略，URL不区分大小写。

**超文本传输协议HTTP：**

一个应用层协议，使用TCP连接。

**超文本标记语言HTML：**

文档结构的标记语言，使用约定标记对页面上的信息、格式进行描述。

### HTTP：

定义了浏览器（即万维网客户进程）怎样向万维网服务器请求万维网文档，以及万维网服务器怎样把万维网文档传送给浏览器。HTTP是面向事务的应用层协议，规定了请求和响应的格式和规则。

**特点：**

早期HTTP是无状态的，服务器并不会记录用户的状态信息。实际应用中，通常使用Cookie跟踪用户活动，服务器会为访问它的用户创建一个识别码，存在数据库中，并在响应报文中添加Set-Cookie首部行给浏览器，浏览器就会管理该服务器主机名和Cookie识别码，下次访问时，就会在请求报文中添加Cookie首部。

**非持续连接方式：**

HTTP/1.0采用非持续连接方式。在该方式下，每次浏览器要请求一个文件都要与服务器建立TCP连接，当收到响应后就立即关闭连接，浏览器通常会建立多个并行的TCP连接同时请求多个对象。每请求一个文档就要2RTT的开销，第三次握手捎带请求报文。若一个网页上有很多引用对象（例如图片等），那么请求每一个对象都需要花费2RTT的时间。

![]({{ '/assets/notes/cn/image103.png' | relative_url }})

**持续连接方式：**

HTTP/1.1采用持续连接方式。在该方式下，万维网服务器在发送响应后仍然保持这条连接，使同一个客户（浏览器）和该服务器可以继续在这条连接上传送后续的HTTP请求报文和响应报文。这并不局限于传送同一个页面上引用的对象，而是只要这些文档都在同一个服务器上就行。非流水线工作时，收到前一个响应才能发起下一个请求。HTTP/1.1的持续连接默认使用流水线方式工作，即浏览器在收到HTTP的响应报文之前就能够连续发送多个请求报文。

无论是持续还是非持续，发的请求报文数量是一样的。

### HTTP报文结构：

HTTP是面向文本的，其报文中的每一个字段都是一些ASCII码串，并且每个字段的长度都是不确定的。
